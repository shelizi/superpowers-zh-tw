# 中文程式碼審查規範

## 概述

國內團隊做 Code Review 常遇到兩個極端：要麼過度客氣導致關鍵問題被放過，要麼照搬西方直白風格讓同事下不來台。本技能幫你找到平衡點——**既不回避問題，又讓人願意接受回饋**。

**核心原則：** 用"建議"代替"命令"，用"提問"代替"否定"，但絕不因為面子而放過 bug。

## 審查回饋的表達方式

### 用建議代替命令

| 避免（命令式） | 推薦（建議式） |
|---------------|---------------|
| 你必須改成 X | 建議考慮用 X，因為 Y |
| 這裡寫錯了 | 這裡可能存在一個問題，是否考慮過 Z 的情況？ |
| 不要用這個方法 | 這個方法在 A 場景下可能有性能問題，可以看看 B 方案 |
| 這段程式碼不行 | 這段邏輯我理解得對嗎？如果輸入為空的話會怎樣？ |

### 用提問代替否定

當你不確定對方意圖時，先問再評：

```
# 好的方式
這裡用 sync 方式讀檔案是出於什麼考慮？如果並發量上來，可能會阻塞事件迴圈。

# 不好的方式
這裡不應該用 sync 方式讀檔案。
```

### 分級標註

統一使用優先級標記，讓作者快速判斷輕重緩急：

- **[必須修復]** — 安全漏洞、資料遺失風險、邏輯錯誤（不修不能合）
- **[建議修改]** — 性能問題、可維護性、缺少校驗（本次或下次迭代修復）
- **[僅供參考]** — 命名優化、風格建議、替代方案（不改也行）
- **[問題]** — 不確定的地方，需要作者解釋意圖

### 審查評論範本

```
[必須修復] SQL 注入風險

第 42 行：使用者輸入直接拼接到 SQL 語句中。

原因：攻擊者可以透過 name 參數注入 `'; DROP TABLE users; --`。

建議：使用參數化查詢：
  db.query('SELECT * FROM users WHERE name = $1', [name])

參考：https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html
```

## 中英混排程式碼註釋規範

### 何時用中文

- **業務邏輯說明** — 用中文解釋業務背景和需求來源
- **複雜演算法註釋** — 用中文寫思路，確保團隊成員都能理解
- **TODO / FIXME** — 用中文描述待辦事項，方便搜尋和追蹤
- **文件註釋（內部專案）** — JSDoc / Javadoc 中的描述文字用中文

```typescript
/**
 * 計算使用者的會員等級折扣
 *
 * 業務規則：
 * - 普通會員 9.5 折
 * - 銀卡會員 9 折
 * - 金卡會員 8.5 折
 * - 鑽石會員 8 折
 *
 * @param level - 會員等級（MemberLevel enum）
 * @param amount - 原始金額（單位：分）
 * @returns 折後金額（單位：分）
 */
function calculateDiscount(level: MemberLevel, amount: number): number {
  // ...
}
```

### 何時用英文

- **變數名、函數名、類別名** — 始終用英文命名，遵循團隊命名規範
- **Git commit message** — 參考下方 commit 規範
- **開源專案註釋** — 面向國際社群的專案，註釋統一用英文
- **錯誤訊息和日誌** — 生產環境的 error message 用英文（避免編碼問題）
- **API 介面文件** — 對外暴露的 API 用英文

### 混排格式要求

```typescript
// 好：中英文之間加空格
// 使用 Redis 快取來減少 MySQL 的查詢壓力

// 壞：中英文之間沒有空格
// 使用Redis快取來減少MySQL的查詢壓力

// 好：技術術語保留英文
// 這裡用 debounce 防抖處理，避免頻繁觸發 API 請求

// 壞：強行翻譯技術術語
// 這裡用防抖動處理，避免頻繁觸發應用程式介面請求
```

## Commit Message 中英雙語格式

### 推薦格式

團隊內部專案使用中文 commit message，採用約定式提交（Conventional Commits）的中文版：

```
<類型>(<範圍>): <簡要描述>

<詳細說明（可選）>

<關聯資訊（可選）>
```

### 類型對照表

| 類型 | 含義 | 範例 |
|------|------|------|
| feat | 新功能 | feat(使用者): 新增手機號登入功能 |
| fix | 修復 Bug | fix(支付): 修復微信支付回調重複處理的問題 |
| docs | 文件變更 | docs: 更新 API 介面文件 |
| style | 程式碼格式 | style: 統一縮排為 2 個空格 |
| refactor | 重構 | refactor(訂單): 拆分訂單服務，提取公共邏輯 |
| perf | 性能優化 | perf(列表): 虛擬捲動優化長列表渲染性能 |
| test | 測試 | test(auth): 補充登入模組單元測試 |
| chore | 建構/工具 | chore: 升級 Node.js 至 v20 |

### 範例

```
fix(支付): 修復支付寶非同步回調簽名校驗失敗的問題

原因：升級 SDK 後簽名演算法從 RSA 變為 RSA2，但回調校驗仍使用舊演算法。
方案：回調處理中同時相容 RSA 和 RSA2 簽名校驗。

Closes #1234
```

### 面向國際社群的專案

如果專案面向國際社群或有外籍成員，commit message 用英文，PR 描述中可附加中文說明：

```
fix(payment): fix Alipay async callback signature verification failure

The SDK upgrade changed the signature algorithm from RSA to RSA2,
but the callback handler still used the old algorithm.

Closes #1234
```

## 常見反模式與對策

### 反模式一：過度客氣

**表現：** 所有評論都是"我覺得可能也許大概好像這裡有個小問題"。

**後果：** 關鍵 bug 被隱藏在一堆委婉語裡，作者根本不知道哪些必須改。

**對策：** 使用分級標註。[必須修復] 就是必須修復，語氣可以溫和，但級別必須準確。

```
# 壞：過度客氣
不知道我理解得對不對，這裡好像可能有一點點並發問題，不過也許我看錯了...

# 好：溫和但清晰
[必須修復] 並發安全問題

這裡的 map 在多個 goroutine 中同時讀寫，會觸發 panic。
建議加 sync.RWMutex，或者換成 sync.Map。

複現方式：加 -race flag 跑測試就能看到。
```

### 反模式二：不敢給高級開發者提意見

**表現：** 高級開發者或 Leader 的程式碼直接 Approve，不仔細看。

**後果：** 程式碼品質雙標，團隊對 Code Review 失去信任。

**對策：** Code Review 對事不對人。可以換個表達方式：

```
# 提問式（適合給資深同事的回饋）
想請教一下，這裡選擇用遞迴而不是迭代，是出於什麼考慮？
我在想如果遞迴深度超過 1000 層會不會有堆疊溢出的風險？

# 學習式
學到了一個新寫法！不過有個小疑問——這裡的型別斷言在執行時不會做檢查，
如果上游資料結構變了，這裡會靜默通過。是否考慮加個 runtime validation？
```

### 反模式三：審查變成風格之爭

**表現：** 大量評論糾結於縮排、空格、花括號位置。

**後果：** 浪費時間，忽略真正的問題。

**對策：** 風格問題交給 ESLint / Prettier / gofmt 等工具自動處理。Code Review 聚焦邏輯、安全、性能。

### 反模式四：只寫"LGTM"

**表現：** 隨手一個 LGTM 就 Approve，沒有實質性審查。

**後果：** Code Review 形同虛設，出了問題沒人兜底。

**對策：** 即使程式碼品質很好，也要寫出你關注了哪些方面：

```
LGTM

審查了以下方面：
- 並發安全：鎖的粒度合理
- 錯誤處理：所有外部呼叫都有 error handling
- 向下相容：新增欄位都有預設值，不影響舊版本

一個小建議 [僅供參考]：第 78 行的變數名 `d` 可以改成 `duration`，更易讀。
```

## 審查流程建議

### 開始審查前

1. **先看 PR 描述**，理解改動的背景和目的
2. **看關聯的 Issue 或需求文件**
3. **先整體瀏覽**，再逐檔案細看

### 審查順序

1. **架構層面** — 方案是否合理？有沒有更好的方式？
2. **正確性** — 邏輯對不對？邊界條件處理了嗎？
3. **安全性** — 有沒有注入、越權、資訊洩露？
4. **性能** — 有沒有 N+1 查詢、記憶體洩漏、不必要的迴圈？
5. **可維護性** — 半年後能看懂嗎？測試覆蓋了嗎？
6. **風格** — 只關注工具無法自動處理的部分

### 給出總結

審查結束後，給一段總結，包括：
- 整體評價（一句話）
- 值得學習的地方（先揚後抑）
- 主要問題列表（按優先級）
- 建議的修改方向

```
總結：整體實作思路清晰，支付回調的冪等處理很到位。

主要問題：
1. [必須修復] 並發寫 map 的問題（2 處）
2. [建議修改] 缺少對空值的校驗（3 處）
3. [僅供參考] 幾個變數命名可以更語義化

建議先修復並發問題，校驗的部分可以本次一起改或者拆到下個迭代。
```

## 檢查清單

在提交審查意見前，確認：

- [ ] 每條評論都標註了優先級
- [ ] [必須修復] 的問題都給出了具體的修復建議
- [ ] 沒有因為面子而跳過關鍵問題
- [ ] 沒有糾結於工具能自動處理的風格問題
- [ ] 對好的程式碼給予了肯定
- [ ] 給出了整體總結
