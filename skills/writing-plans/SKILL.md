# 撰寫計畫

## 概述

撰寫全面的實作計畫，假設工程師對我們的程式碼庫零上下文，且品味存疑。記錄他們需要知道的一切：每個任務要修改哪些檔案、程式碼、測試、可能需要查閱的文件、如何測試。將整個計畫拆成小步驟任務。DRY。YAGNI。TDD。頻繁 commit。

假設他們是有經驗的開發者，但我們的工具鏈和問題領域幾乎一無所知。假設他們不太擅長測試設計。

**開始時宣布：** "我正在使用 writing-plans 技能建立實作計畫。"

**上下文：** 此技能應在專用 worktree 中執行（由 brainstorming 技能建立）。

**計畫儲存位置：** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- （使用者對計畫位置的偏好優先於此預設值）

## 範圍檢查

如果規格涵蓋了多個獨立子系統，它應該在腦力激盪階段就被拆分為子專案規格。如果沒有，建議將其拆分為獨立的計畫——每個子系統一個。每個計畫應該能獨立產出可工作、可測試的軟體。

## 檔案結構

在定義任務之前，先列出將要建立或修改的檔案以及每個檔案的職責。這是鎖定分解决策的地方。

- 設計邊界清晰、介面定義良好的單元。每個檔案應有一個明確的職責。
- 你對能一次放入上下文的程式碼推理得最好，檔案越專注你的編輯越可靠。優先選擇小而專注的檔案，而非承擔過多功能的大檔案。
- 一起變更的檔案應放在一起。按職責拆分，而非按技術層級拆分。
- 在現有程式碼庫中，遵循已有模式。如果程式碼庫使用大檔案，不要單方面重構——但如果你正在修改的檔案已經變得難以管理，在計畫中包含拆分是合理的。

此結構決定了任務分解。每個任務應產出獨立的、有意義的變更。

## 小步驟任務粒度

**每步是一個操作（2-5 分鐘）：**
- "撰寫失敗的測試" - 一步
- "執行它確認失敗" - 一步
- "實作最少程式碼讓測試通過" - 一步
- "執行測試確認通過" - 一步
- "Commit" - 一步

## 計畫文件頭部

**每個計畫必須以此頭部開始：**

```markdown
# [功能名稱] 實作計畫

> **面向 AI 代理的工作者：** 必需子技能：使用 superpowers:subagent-driven-development（推薦）或 superpowers:executing-plans 逐任務實作此計畫。步驟使用複選框（`- [ ]`）語法來追蹤進度。

**目標：** [一句話描述要建構什麼]

**架構：** [2-3 句話描述方案]

**技術棧：** [關鍵技術/庫]

---
```

## 任務結構

````markdown
### 任務 N：[元件名稱]

**檔案：**
- 建立：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 測試：`tests/exact/path/to/test.py`

- [ ] **步驟 1：撰寫失敗的測試**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步驟 2：執行測試驗證失敗**

執行：`pytest tests/path/test.py::test_name -v`
預期：FAIL，報錯 "function not defined"

- [ ] **步驟 3：撰寫最少實作程式碼**

```python
def function(input):
    return expected
```

- [ ] **步驟 4：執行測試驗證通過**

執行：`pytest tests/path/test.py::test_name -v`
預期：PASS

- [ ] **步驟 5：Commit**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 禁止佔位符

每個步驟都必須包含工程師需要的實際內容。以下是**計畫缺陷**——絕不要寫出來：
- "待定"、"TODO"、"後續實作"、"補充細節"
- "新增適當的錯誤處理" / "新增驗證" / "處理邊界情況"
- "為上述程式碼撰寫測試"（沒有實際測試程式碼）
- "類似任務 N"（重複程式碼——工程師可能不按順序閱讀任務）
- 只描述做什麼而不展示怎麼做的步驟（程式碼步驟必須有程式碼區塊）
- 引用了未在任何任務中定義的類型、函數或方法

## 注意事項
- 始終使用精確的檔案路徑
- 每個步驟都包含完整程式碼——如果步驟涉及程式碼變更，就展示程式碼
- 精確的命令和預期輸出
- DRY、YAGNI、TDD、頻繁 commit

## 自檢

撰寫完整計畫後，以全新視角審視規格並對照檢查計畫。這是你自己執行的檢查清單——不是子代理排程。

**1. 規格覆蓋度：** 瀏覽規格中的每個章節/需求。你能指出實作它的任務嗎？列出所有遺漏。

**2. 佔位符掃描：** 搜尋計畫中的紅旗——上方"禁止佔位符"章節中的任何模式。修復它們。

**3. 類型一致性：** 後續任務中使用的類型、方法簽名和屬性名是否與前面任務中定義的一致？任務 3 中叫 `clearLayers()` 但任務 7 中叫 `clearFullLayers()` 就是 bug。

如果發現問題，直接內聯修復。無需重新審查——修好繼續推進。如果發現規格中的需求沒有對應任務，就新增任務。

## 執行交接

儲存計畫後，提供執行選項：

**"計畫已完成並儲存到 `docs/superpowers/plans/<filename>.md`。兩種執行方式：**

**1. 子代理驅動（推薦）** - 每個任務排程一個新的子代理，任務間進行審查，快速迭代

**2. 內聯執行** - 在當前會話中使用 executing-plans 執行任務，批次執行並設有檢查點

**選擇哪種方式？**

**如果選擇子代理驅動：**
- **必需子技能：** 使用 superpowers:subagent-driven-development
- 每個任務一個新子代理 + 兩階段審查

**如果選擇內聯執行：**
- **必需子技能：** 使用 superpowers:executing-plans
- 批次執行並設有檢查點供審查
