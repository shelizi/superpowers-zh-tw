# 執行計畫

## 概述

載入計畫，批判性審查，執行所有任務，完成後報告。

**開始時宣布：** "我正在使用 executing-plans 技能來實作此計畫。"

**注意：** 告訴你的人類夥伴，Superpowers 在有子代理支援時效果好得多。如果在支援子代理的平台上執行（如 Claude Code 或 Codex），其工作品質會顯著提高。如果子代理可用，請使用 superpowers:subagent-driven-development 而非此技能。

## 流程

### 步驟 1：載入並審查計畫

1. 讀取計畫檔案
2. 批判性審查——識別計畫中的任何問題或疑慮
3. 如果有疑慮：在開始之前向你的人類夥伴提出
4. 如果沒有疑慮：建立 TodoWrite 並繼續

**審查時重點檢查：**
- 步驟之間是否有相依性遺漏？（A 相依 B，但 B 排在 A 之後）
- 驗證條件是否明確？（"確認可用"不算，"執行 `npm test` 全部通過"才算）
- 是否有隱含的環境假設？（Node 版本、資料庫連接、API Key）

**審查範例：**
```
計畫檔案：docs/plan.md
任務清單：5 個任務

審查發現：
- 任務 3（新增資料庫遷移）應在任務 2（撰寫資料模型）之後，順序正確 ✓
- 任務 4 的驗證條件寫的是"確認功能正常"→ 需釐清：具體跑什麼測試？
- 計畫未提及 Python 版本要求 → 需確認

向夥伴提出：
"計畫整體可執行。有兩個問題：(1) 任務 4 的驗證條件不夠具體，建議改為
'執行 pytest tests/test_api.py 全部通過'；(2) 需要確認 Python 版本要求。"
```

### 步驟 2：執行任務

對於每個任務：

1. **標記為進行中** — 更新 TodoWrite
2. **理解目標** — 重讀任務描述，明確完成標準
3. **執行實作** — 嚴格按照計畫步驟執行（計畫已有小步驟）
4. **執行驗證** — 按要求執行測試或檢查
5. **提交變更** — 每完成一個任務提交一次，commit message 引用任務編號
6. **標記為已完成** — 更新 TodoWrite

**每個任務的節奏：**
```
--- 任務 2/5：新增使用者驗證 ---
[标记进行中]

目標：為 /api/users 新增輸入驗證
完成標準：所有驗證測試通過，無效輸入返回 400

[實作]
- 新增 validateUser() 中介軟體
- 撰寫 3 個驗證規則（email 格式、密碼強度、使用者名稱長度）

[驗證]
$ npm test -- --grep "validation"
  ✓ 拒绝无效 email (12ms)
  ✓ 拒绝弱密码 (8ms)
  ✓ 拒绝过长用户名 (5ms)
  3 passing

[提交]
$ git add src/middleware/validate.js tests/validation.test.js
$ git commit -m "feat: 新增使用者輸入驗證（任務 2/5）"

[标记完成]
--- 任务 2/5 完成 ---
```

**批次審查檢查點：**
- 每完成 3 個任務後，暫停回顧：整體方向還對嗎？有沒有偏離計畫？
- 如果發現前面的實作有問題，先修復再繼續，不要帶著問題往下走

### 步驟 3：處理常見異常

**測試失敗：**
1. 讀錯誤訊息，定位失敗原因
2. 區分：是實作 bug？還是測試本身有問題？還是計畫描述有誤？
3. 實作 bug → 修復並重跑
4. 測試有問題 → 修復測試，向夥伴說明
5. 計畫有誤 → 停下來，向夥伴報告並建議修正

**相依性缺失：**
```
任務 3 需要 Redis 連接，但計畫中沒有提及 Redis 設定。
→ 停止執行
→ 向夥伴報告："任務 3 需要 Redis，計畫中未包含設定步驟。
   建議：在任務 3 前插入 '設定 Redis 連接' 步驟。"
```

**指令不清：**
- 不要猜測意圖，不要"合理推斷"
- 列出你的理解和困惑，讓夥伴釐清
- 等待回覆後再繼續

### 步驟 4：完成開發

所有任務完成並驗證後：
- 宣布："我正在使用 finishing-a-development-branch 技能來完成此工作。"
- **必需子技能：** 使用 superpowers:finishing-a-development-branch
- 按照該技能的指引驗證測試、展示選項、執行選擇

**完成報告範本：**
```
## 執行報告

**計畫：** docs/plan.md
**分支：** feature/user-validation
**任務：** 5/5 已完成

### 完成的任務
1. ✅ 初始化專案結構
2. ✅ 新增使用者驗證
3. ✅ 新增資料庫遷移
4. ✅ 實作 API 端點
5. ✅ 新增整合測試

### 驗證結果
- 單元測試：23/23 通過
- 整合測試：8/8 通過
- lint 檢查：0 個警告

### 偏離計畫的地方
- 任務 3：Redis 設定從 env 改為 config.yaml（經夥伴同意）

### 下一步
按 finishing-a-development-branch 技能處理合併/PR
```

## 何時停下來求助

**在以下情況立即停止執行：**
- 遇到阻塞（缺少相依性、測試失敗、指令不清）
- 計畫有嚴重缺陷導致無法開始
- 你不理解某條指令
- 驗證反覆失敗（同一測試失敗 2 次以上）

**不確定時就問，不要猜測。**

## 何時回到之前的步驟

**回到審查（步驟 1）當：**
- 夥伴根據你的回饋更新了計畫
- 根本性的方案需要重新考慮

**不要硬闖阻塞** — 停下來問。

## 注意事項
- 先批判性審查計畫
- 嚴格按照計畫步驟執行
- 不要跳過驗證
- 每個任務單獨提交，commit message 引用任務編號
- 計畫要求時引用相應技能
- 遇到阻塞時停下來，不要猜測
- 未經使用者明確同意，絕不在 main/master 分支上開始實作

## 整合

**必需的工作流技能：**
- **superpowers:using-git-worktrees** - 必需：開始前建立隔離的工作空間
- **superpowers:writing-plans** - 建立此技能要執行的計畫
- **superpowers:finishing-a-development-branch** - 所有任務完成後收尾開發
