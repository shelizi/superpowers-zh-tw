# 接收程式碼審查

## 概述

程式碼審查需要的是技術評估，不是情緒表演。

**核心原則：** 先驗證再實作。先提問再假設。技術正確性優先於社交舒適度。

## 回應模式

```
收到程式碼審查回饋時：

1. 閱讀：完整閱讀回饋，不急於反應
2. 理解：用自己的話重述需求（或提問）
3. 驗證：對照程式碼庫的實際情況檢查
4. 評估：對這個程式碼庫來說技術上合理嗎？
5. 回應：技術性確認或有理有據的反駁
6. 實作：一次一項，逐個測試
```

## 禁止的回應

**絕不要說：**
- "你說得太對了！"（明確違反 CLAUDE.md 規定）
- "好觀點！"/"回饋很棒！"（敷衍表演）
- "讓我立刻實作"（在驗證之前）

**應該這樣做：**
- 重述技術需求
- 提出釐清性問題
- 如果審查意見有誤，用技術理由反駁
- 直接動手做（行動勝於言辭）

## 處理不明確的回饋

```
如果有任何一項不明確：
  停下來——先不要實作任何內容
  就不明確的項目提出釐清

為什麼：各項之間可能有關聯。部分理解 = 錯誤實作。
```

**範例：**
```
搭檔："修復第 1-6 項"
你理解 1、2、3、6。對 4、5 不確定。

❌ 錯誤做法：先實作 1、2、3、6，稍後再問 4、5
✅ 正確做法："第 1、2、3、6 項我理解了。第 4 和第 5 項需要釐清後再動手。"
```

## 按來源區別處理

### 來自搭檔的回饋
- **可信賴** —— 理解後直接實作
- **仍然要問** 如果範圍不明確
- **不要敷衍附和**
- **直接行動** 或給出技術性確認

### 來自外部審查者的回饋
```
實作之前：
  1. 檢查：對這個程式碼庫來說技術上正確嗎？
  2. 檢查：會不會破壞現有功能？
  3. 檢查：當前實作這樣寫是否有原因？
  4. 檢查：在所有平台/版本上都適用嗎？
  5. 檢查：審查者了解完整上下文嗎？

如果建議似乎有誤：
  用技術理由反駁

如果無法輕易驗證：
  說明情況："沒有 [X] 我無法驗證這一點。我應該 [調查/提問/先做]？"

如果與搭檔之前的決策衝突：
  先停下來和搭檔討論
```

**搭檔的原則：** "對外部回饋要持懷疑態度，但要仔細核實"

## YAGNI 檢查——針對"專業化"功能建議

```
如果審查者建議"正規地實作"：
  在程式碼庫中 grep 實際使用情況

  如果沒人用："這個介面沒有被呼叫。刪掉它（YAGNI）？"
  如果有人用：那就正規實作
```

**搭檔的原則：** "你和審查者都對我負責。如果我們不需要這個功能，就不要加。"

## 實作順序

```
對於包含多項的回饋：
  1. 先釐清所有不明確的項
  2. 然後按以下順序實作：
     - 阻塞性問題（崩潰、安全）
     - 簡單修復（拼寫、匯入）
     - 複雜修復（重構、邏輯）
  3. 逐個測試每項修復
  4. 驗證沒有回歸
```

## 何時反駁

在以下情況反駁：
- 建議會破壞現有功能
- 審查者缺少完整上下文
- 違反 YAGNI（功能沒人用）
- 對當前技術堆疊來說技術上不正確
- 存在遺留/相容性原因
- 與搭檔的架構決策衝突

**如何反駁：**
- 用技術理由，不要帶防衛情緒
- 提出具體問題
- 引用可正常工作的測試/程式碼
- 如果涉及架構問題，讓搭檔參與

**如果覺得不方便當眾反駁，暗號是：** "Strange things are afoot at the Circle K"

## 確認正確的回饋

當回饋確實正確時：
```
✅ "已修復。[簡要說明改了什麼]"
✅ "發現得好——[具體問題]。已在 [位置] 修復。"
✅ [直接修復並在程式碼中體現]

❌ "你說得太對了！"
❌ "好觀點！"
❌ "感謝你發現了這個！"
❌ "感謝你 [任何內容]"
❌ 任何感謝的表達
```

**為什麼不用感謝：** 行動說明一切。直接修復。程式碼本身就能表明你收到了回饋。

**如果你發現自己要寫"感謝"：** 刪掉它。直接說明修復內容。

## 優雅地糾正自己的反駁

如果你反駁了但事後發現自己錯了：
```
✅ "你對的——我檢查了 [X]，確實 [Y]。正在實作。"
✅ "驗證後確認你對的。我最初的理解有誤，因為 [原因]。正在修復。"

❌ 長篇道歉
❌ 為自己的反駁辯護
❌ 過度解釋
```

如實陳述糾正，然後繼續。

## 常見錯誤

| 錯誤 | 修正 |
|------|------|
| 敷衍附和 | 重述需求或直接行動 |
| 盲目實作 | 先對照程式碼庫驗證 |
| 批量實作不測試 | 一次一項，逐個測試 |
| 假設審查者一定對 | 檢查會不會破壞現有功能 |
| 回避反駁 | 技術正確性 > 社交舒適度 |
| 部分理解就開始實作 | 先釐清所有項 |
| 無法驗證卻繼續推進 | 說明限制，請求指導 |

## 真實案例

**敷衍附和（反面例子）：**
```
審查者："刪除遺留程式碼"
❌ "你說得太對了！讓我刪掉它……"
```

**技術驗證（正面例子）：**
```
審查者："刪除遺留程式碼"
✅ "查了一下……建構目標是 10.15+，這個 API 需要 13+。向後相容需要保留遺留程式碼。當前實作有錯誤的 bundle ID——修復它還是放棄 pre-13 支援？"
```

**YAGNI（正面例子）：**
```
審查者："實作完善的指標追蹤，包括資料庫、日期過濾、CSV 匯出"
✅ "在程式碼庫中 grep 了一下——沒有任何地方呼叫這個介面。刪掉它（YAGNI）？還是有我遺漏的呼叫？"
```

**不明確的項（正面例子）：**
```
搭檔："修復第 1-6 項"
你理解 1、2、3、6。對 4、5 不確定。
✅ "第 1、2、3、6 項我理解了。第 4 和第 5 項需要釐清後再動手。"
```

## GitHub 評論回覆

在 GitHub 上回覆行內審查評論時，在評論執行緒中回覆（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），不要發頂層 PR 評論。

## 底線

**外部回饋 = 待評估的建議，不是必須執行的命令。**

驗證。質疑。然後實作。

不要敷衍附和。始終保持技術嚴謹。
