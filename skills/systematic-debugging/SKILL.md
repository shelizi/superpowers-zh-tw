# 系統化除錯

## 概述

隨意修復既浪費時間又會引入新 bug。草率的補丁只會掩蓋深層問題。

**核心原則：** 在嘗試修復之前，務必先找到根本原因。只修症狀就是失敗。

**敷衍走流程等於違背除錯的精神。**

## 鐵律

```
不做根因調查，不許提修復方案
```

如果你還沒完成第一階段，就不能提出修復方案。

## 何時使用

用於任何技術問題：
- 測試失敗
- 生產環境 bug
- 異常行為
- 效能問題
- 建構失敗
- 整合問題

**尤其在以下情況必須使用：**
- 時間緊迫（緊急情況最容易讓人猜測式修復）
- 覺得"一個小修改"就能搞定
- 已經嘗試了多種修復
- 上一次修復沒有生效
- 你沒有完全理解問題

**以下情況也不要跳過：**
- 問題看起來很簡單（簡單的 bug 也有根本原因）
- 你很趕時間（越急越容易返工）
- 領導要求立刻修好（系統化除錯比反覆嘗試更快）

## 四個階段

你必須完成每個階段後才能進入下一個。

### 第一階段：根因調查

**在嘗試任何修復之前：**

1. **仔細閱讀錯誤訊息**
   - 不要跳過錯誤或警告
   - 它們往往直接包含解決方案
   - 完整閱讀堆疊追蹤
   - 記下行號、檔案路徑、錯誤碼

2. **穩定重現**
   - 你能可靠地觸發它嗎？
   - 具體的重現步驟是什麼？
   - 每次都能重現嗎？
   - 如果無法重現 → 收集更多資料，不要猜測

3. **檢查近期變更**
   - 什麼變更可能導致了這個問題？
   - git diff、最近的提交
   - 新相依性、設定變更
   - 環境差異

4. **在多元件系統中收集證據**

   **當系統有多個元件時（CI → 建構 → 簽名，API → 服務 → 資料庫）：**

   **在提出修復方案之前，先新增診斷埋點：**
   ```
   對每個元件邊界：
     - 記錄進入元件的資料
     - 記錄離開元件的資料
     - 驗證環境/設定的傳遞
     - 檢查每一層的狀態

   執行一次以收集證據，確定斷裂點在哪裡
   然後分析證據，定位故障元件
   然後針對該元件深入調查
   ```

   **範例（多層系統）：**
   ```bash
   # 第 1 層：工作流
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 層：建構腳本
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第 3 層：簽名腳本
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # 第 4 層：實際簽名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **由此可以看出：** 哪一層出了問題（secrets → workflow ✓, workflow → build ✗）

5. **追蹤資料流**

   **當錯誤發生在呼叫堆疊深處時：**

   參見本目錄下的 `root-cause-tracing.md`，了解完整的反向追蹤技術。

   **簡要版本：**
   - 錯誤值從哪裡產生的？
   - 誰用錯誤值呼叫了這裡？
   - 持續向上追蹤直到找到源頭
   - 在源頭修復，而不是在症狀處修復

### 第二階段：模式分析

**先找到模式，再修復：**

1. **找到可正常工作的範例**
   - 在同一程式碼庫中找到類似的正常程式碼
   - 有什麼正常的程式碼與出問題的程式碼相似？

2. **與參考實作對比**
   - 如果是實作某個模式，完整閱讀參考實作
   - 不要略讀——逐行閱讀
   - 在應用之前徹底理解該模式

3. **識別差異**
   - 正常程式碼和出問題的程式碼之間有什麼不同？
   - 列出每一個差異，無論多小
   - 不要假設"那不可能有影響"

4. **理解相依性關係**
   - 這個功能需要哪些其他元件？
   - 需要哪些設定、設定、環境？
   - 它有哪些隱含假設？

### 第三階段：假設與驗證

**科學方法：**

1. **提出單一假設**
   - 清晰地陳述："我認為 X 是根本原因，因為 Y"
   - 寫下來
   - 要具體，不要含糊

2. **最小化測試**
   - 做出最小的改動來驗證假設
   - 每次只改一個變數
   - 不要同時修復多個問題

3. **繼續之前先驗證**
   - 生效了？是 → 進入第四階段
   - 沒生效？提出新假設
   - 不要在上面疊加更多修復

4. **當你不確定時**
   - 說"我不理解 X"
   - 不要假裝自己知道
   - 尋求幫助
   - 做更多調研

### 第四階段：實作

**修復根本原因，而非症狀：**

1. **建立失敗的測試用例**
   - 最簡化的重現
   - 盡可能用自動化測試
   - 沒有測試框架就寫一次性測試腳本
   - 修復前必須先有測試
   - 使用 `superpowers:test-driven-development` 技能來撰寫規範的失敗測試

2. **實作單一修復**
   - 修復已定位的根本原因
   - 每次只改一處
   - 不做"順便改改"的優化
   - 不捆綁重構

3. **驗證修復**
   - 測試現在通過了嗎？
   - 其他測試沒有被破壞吧？
   - 問題真的解決了嗎？

4. **如果修復不起作用**
   - 停下來
   - 數一數：你已經嘗試了幾次修復？
   - 少於 3 次：回到第一階段，用新資訊重新分析
   - **3 次或以上：停下來質疑架構（見下方第 5 步）**
   - 沒有經過架構討論，不要嘗試第 4 次修復

5. **如果 3 次以上修復都失敗了：質疑架構**

   **以下模式表明存在架構問題：**
   - 每次修復都暴露出新的共享狀態/耦合/其他位置的問題
   - 修復需要"大規模重構"才能實作
   - 每次修復都在其他地方產生新的症狀

   **停下來質疑根本性問題：**
   - 這個模式從根本上合理嗎？
   - 我們是不是在"慣性驅動"下堅持了錯誤方案？
   - 應該重構架構還是繼續修補症狀？

   **在嘗試更多修復之前，和你的搭檔討論**

   這不是假設失敗——這是架構有誤。

## 紅線——停下來，按流程走

如果你發現自己在想：
- "先臨時修一下，以後再排查"
- "試著改改 X 看看行不行"
- "一次性改多個地方，跑測試看看"
- "跳過測試，我手動驗證"
- "大概是 X 的問題，讓我修一下"
- "我不完全理解，但這應該能行"
- "模式說的是 X，但我換個方式用"
- "主要問題有這些：[未經調查就列出修復方案]"
- 沒有追蹤資料流就提出解決方案
- **"再試一次修復"（已經嘗試了 2 次以上）**
- **每次修復都暴露出不同地方的新問題**

**以上這些都意味著：停下來。回到第一階段。**

**如果 3 次以上修復都失敗了：** 質疑架構（見第四階段第 5 步）

## 搭檔發出的訊號——說明你的方法不對

**留意這些提醒：**
- "難道不是這樣嗎？"——你在沒有驗證的情況下做了假設
- "它能告訴我們……嗎？"——你應該先收集證據
- "別猜了"——你在沒有理解的情況下提出修復
- "深入想想"——要質疑根本性問題，而不只是症狀
- "我們卡住了？"（沮喪的語氣）——你的方法沒有奏效

**當你看到這些訊號時：** 停下來。回到第一階段。

## 常見藉口

| 借口 | 现实 |
|------|------|
| "問題很簡單，不需要走流程" | 簡單問題也有根本原因。對於簡單 bug，流程很快就能走完。 |
| "緊急情況，沒時間走流程" | 系統化除錯比反覆猜測式修復更快。 |
| "先試一下，再排查" | 第一次修復就定下了基調。從一開始就做對。 |
| "確認修復有效後再寫測試" | 沒有測試的修復留不住。先寫測試才能證明修復有效。 |
| "一次修多個問題省時間" | 無法隔離哪個生效了。還會引入新 bug。 |
| "參考實作太長了，我自己改改" | 一知半解必然出 bug。完整閱讀。 |
| "我看出了問題，讓我修一下" | 看到症狀 ≠ 理解根因。 |
| "再試一次"（在 2 次以上失敗後） | 3 次以上失敗 = 架構問題。質疑模式，不要繼續修。 |

## 速查表

| 階段 | 關鍵活動 | 通過標準 |
|------|---------|---------|
| **1. 根因** | 閱讀錯誤、重現、檢查變更、收集證據 | 理解了什麼出了問題以及為什麼 |
| **2. 模式** | 找到正常範例、對比 | 識別出差異 |
| **3. 假設** | 提出理論、最小化驗證 | 假設被驗證或產生新假設 |
| **4. 實作** | 建立測試、修復、驗證 | bug 已修復，測試通過 |

## 當流程顯示"找不到根因"

如果系統化排查後發現問題確實是環境相關、時序相關或外部因素導致的：

1. 你已經完成了流程
2. 記錄你排查了什麼
3. 實作適當的處理措施（重試、逾時、錯誤提示）
4. 新增監控/日誌以便後續排查

**但是：** 95% 的"找不到根因"其實是排查不充分。

## 輔助技術

以下技術是系統化除錯的組成部分，可在本目錄中找到：

- **`root-cause-tracing.md`** - 沿呼叫堆疊反向追蹤 bug，找到最初的觸發點
- **`defense-in-depth.md`** - 找到根因後，在多個層級新增校驗
- **`condition-based-waiting.md`** - 用條件輪詢替代硬編碼等待時間

**相關技能：**
- **superpowers:test-driven-development** - 用於建立失敗測試用例（第四階段，第 1 步）
- **superpowers:verification-before-completion** - 在聲稱成功之前驗證修復確實有效

## 實際效果

除錯實踐中的資料：
- 系統化方法：15-30 分鐘修復
- 隨意修復方法：2-3 小時反覆折騰
- 一次修復成功率：95% vs 40%
- 引入新 bug：幾乎為零 vs 經常發生
